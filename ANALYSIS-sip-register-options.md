# Анализ: PBX extension → браузер (REGISTER)

Документ фиксирует сравнение вариантов «заменить SIP-телефон сотрудника веб-клиентом», когда на PBX у каждого есть extension и нужен именно **SIP REGISTER** (как у устройства).

Связанные материалы: [`PLAN-collective-calls.md`](./PLAN-collective-calls.md), текущий стек на LiveKit.

---

## 1. Цель

```
Сейчас:  Звонок на 101 → PBX → SIP-телефон / софтфон
Нужно:   Звонок на 101 → PBX → веб-клиент сотрудника (коллектив)
```

Ограничение: PBX ожидает **REGISTER** на extension (как от трубки), а не только static SIP trunk.

---

## 2. Важное про LiveKit

**LiveKit SIP не умеет исходящий SIP REGISTER** на PBX.  
Поддерживается модель trunk / auth на **INVITE**.

Поэтому «чистый LiveKit SIP вместо трубки» в схеме REGISTER **не подходит**. Нужен либо другой стек, либо прослойка, которая REGISTER’ится сама.

---

## 3. Кандидаты

### A. Janus SIP plugin

```
Браузер ── WebRTC + Janus API (не SIP) ──▶ Janus (SIP plugin)
                                              │
                                              └── SIP REGISTER / INVITE ──▶ PBX
```

- В браузере обычно **не sip.js**, а клиент Janus (`janus.js` / Janus WebSocket API).
- **REGISTER на PBX делает Janus** от имени сотрудника.
- Браузер с PBX по SIP не говорит.

### B. sip.js напрямую в браузере

```
Браузер (sip.js + WebRTC) ── SIP REGISTER / INVITE (часто WSS) ──▶ PBX
```

- Браузер сам регистрируется как SIP UA.
- Janus/LiveKit для телефонии не обязательны.

### C. LiveKit + B2BUA (регистратор)

```
B2BUA / тонкий Asterisk ── REGISTER ──▶ PBX
         │
         └── INVITE / bridge ──▶ LiveKit SIP / room ──▶ браузер (WebRTC)
```

- REGISTER делает стабильный серверный шлюз.
- Браузер без SIP; коллектив/SFU остаётся на LiveKit (текущий MVP).

---

## 4. Сравнение по критериям

| Критерий | Janus (A) | sip.js напрямую (B) | LiveKit + B2BUA (C) |
|----------|-----------|---------------------|---------------------|
| REGISTER «как трубка» | Отлично | Отлично | Отлично (на шлюзе) |
| NAT / файрвол у клиента | Лучше: медиа на Janus | Хуже: чаще нужен правильный WSS/ICE/TURN у PBX | Лучше: медиа на LiveKit/шлюзе |
| Секреты SIP на PBX | На сервере (Janus) | Ближе к клиенту — выше риск | На сервере (B2BUA) |
| Стабильность регистрации | Высокая (сервер держит N REGISTER) | Ниже: вкладка, sleep, смена сети | Высокая, если шлюз always-on |
| Несколько вкладок / устройств | Проще централизовать | Конфликт Contact на одном extension | Централизовано |
| Входящие с PBX | Надёжно, пока Janus up | Только пока вкладка открыта и зарегистрирована | Надёжно, пока B2BUA up |
| Видео-конференции коллектива | Отдельный Videoroom / второй стек | Не про SFU | Уже есть (текущий MVP) |
| Сложность эксплуатации | Средняя (Janus + SIP plugin) | Средняя/высокая на стыке PBX↔WebSocket SIP | Средняя (+ ещё один сервис) |
| Зрелость под «заменить телефон» | Очень высокая | Высокая для softphone | Высокая как telephony bridge |

---

## 5. Разбор вариантов

### 5.1. sip.js напрямую

**Плюсы**

- Меньше звеньев в цепочке.
- Классический softphone.
- PBX видит «настоящий» UA.

**Минусы для продакшена коллектива**

- Регистрация живёт, пока открыт браузер.
- SIP-пароль ближе к клиенту.
- С ноутбука за NAT/VPN чаще проблемы, чем у server-side REGISTER.
- Один extension — две вкладки → типичные проблемы Contact / fork.

**Вывод:** ок для простого softphone «один человек — одна вкладка».  
Для «замена рабочего телефона всему коллективу» — **слабее**.

### 5.2. Janus

**Плюсы**

- Прямо закрывает сценарий «устройство = сервер, который REGISTER’ится».
- Медиа и SIP якорятся на сервере → предсказуее.
- Credentials можно не отдавать в браузер целиком (сессия через backend → Janus).
- Входящие устойчивее, если register держит сервер, а не только жизнь вкладки.

**Минусы**

- Это не LiveKit: комнаты/SFU/UX другие.
- Текущий MVP на LiveKit почти не переиспользуется.
- Janus = gateway; продукт «коллектив» всё равно пишется сами.

**Вывод:** самый работоспособный вариант, если **главный** кейс — телефония PBX с REGISTER.

### 5.3. LiveKit + B2BUA

**Плюсы**

- Сохраняется текущий коллектив / 1:1 / SFU.
- REGISTER делает стабильный сервер.
- Браузер без SIP.

**Минусы**

- Два контура (LiveKit + шлюз).
- Отладка SIP сложнее.
- Community B2BUA ≠ «официальный» продукт; часто надёжнее тонкий Asterisk/FreeSWITCH только как регистратор.

**Вывод:** лучший **компромисс**, если коллектив на LiveKit уже ценен, а PBX — следующий слой.

---

## 6. Что работоспособнее

Зависит от приоритета.

| Приоритет | Рекомендация |
|-----------|--------------|
| «Как телефон на extension, REGISTER обязателен» | **Janus** (или PBX WebRTC в той же server-side модели) |
| «Коллектив + видео как сейчас + PBX потом» | **LiveKit + B2BUA / маленький Asterisk** |
| Быстрый softphone, мало пользователей | **sip.js** (прототип, не лучший prod для офиса) |

### Практическая рекомендация для офисной замены трубок

**Наиболее работоспособен server-side REGISTER (Janus или B2BUA), не sip.js в браузере.**

Между Janus и LiveKit+B2BUA:

- Минимум сущностей вокруг телефонии, готовы сменить/добавить медиастек → **Janus**.
- Оставить текущий LiveKit-коллектив → **LiveKit + регистратор (B2BUA/Asterisk)** — обычно прагматичнее, чем всё переносить на Janus.

**sip.js** — самый простой в демо, но **наименее устойчивый** как замена рабочего устройства на парк сотрудников.

---

## 7. Уточнение про «sip.js → Janus»

Частая путаница: «sip.js в браузере подключается к Janus, а Janus регистрируется на PBX».

**Правильно для Janus SIP plugin:**

- браузер ↔ Janus по **Janus API + WebRTC**;
- SIP (включая REGISTER) — только **Janus ↔ PBX**.

`sip.js` — отдельный путь без Janus.

---

## 8. Итог одной фразой

Для обязательного REGISTER на существующую PBX: **не LiveKit SIP alone**; выбирайте **Janus** (телефония-first) или **LiveKit + server-side регистратор** (коллектив-first). **sip.js** — только лёгкий softphone, не лучшая замена парка трубок.
